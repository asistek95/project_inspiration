01 — SYSTEMARCHITEKTUR
🏗

Gesamtüberblick

Klarblick ist eine Multi-Tenant SaaS-Plattform auf Basis von Next.js 14 (App Router) und Supabase. Alle Komponenten kommunizieren über definierte API-Routes — kein direkter Datenbank-Zugriff vom Frontend. Architektur-Prinzip: localStorage-first, Cloud im Hintergrund — jede Schreibaktion landet sofort lokal, Supabase-Sync läuft asynchron (bgSync) und blockiert nie das Speichern.

Eingangskanäle
📁 Datei-Upload (PDF/JPG/PNG/HEIC)
📷 Kamera/Foto (Browser MediaDevices)
💬 WhatsApp-Foto (Twilio Webhook)
✉️ E-Mail-Anhang (Postmark Webhook)
🌐 Web-Interface (Next.js UI)
Next.js Backend-Routen
/api/ocr — Claude Vision Extraktion
/api/whatsapp — Webhook, HMAC-Validierung
/api/email/webhook — Postmark JSON Parser
/api/ai/chat — Steuerberater-Assistent
Middleware — Auth-Guard, Supabase JWT
Daten & Dienste
🤖 Claude Vision API (claude-sonnet-4-6)
🗄 Supabase DB (PostgreSQL, RLS)
🔐 Supabase Auth (JWT, Email OTP)
📦 Supabase Storage (Belegdateien, 7 Jahre)
📋 Audit-Log (GoBD, §132 BAO)
Sync-Architektur neu
localStorage = Source of Truth (sofort)
bgSync() nach jedem Upsert/Update/Delete
Pending-Queue bei Offline-Zustand
initFromSupabase() beim ersten Login
Designentscheidung: Kein Datenverlust durch Netzwerkausfälle — der kritische Pfad (Beleg speichern) hängt nie von Supabase-Erreichbarkeit ab. Storage-Upload und Cloud-Sync laufen strikt im Hintergrund (.then/.catch), niemals blockierend.

4-Stufen-Klassifikation: Eingang vs. Ausgang neu dokumentiert

Statt einer einfachen ATU-Prüfung läuft die Erkennung über vier Stufen mit fallender Konfidenz:

StufeBedingungErgebnisConfidence
1aATU des Ausstellers = Unternehmens-ATUAusgang0,97
1bATU des Empfängers = Unternehmens-ATUEingang0,97
2aFirmenname Aussteller ≈ Unternehmensname (Fuzzy)Ausgang0,90
2bFirmenname Empfänger ≈ Unternehmensname (Fuzzy)Eingang0,90
3Belegart-Heuristik (Kassenbon, Tankbeleg → immer Eingang)Eingang/neutral0,85
4OCR-Rateversuch ohne klaren Trefferunknown0,60

Das Ergebnis wird inkl. Begründungstext (invoice_type_reason) gespeichert und in der UI für den Steuerberater transparent angezeigt — z. B. „Kassenbon → immer Einkaufsbeleg (Eingang)".

Richtungsabhängige Kategorien neu

03 — MULTI-TENANT
🏢

Datentrennung & Rollen

Klarblick ist eine Multi-Tenant-Plattform — jeder Betrieb (Tenant) sieht ausschließlich seine eigenen Daten. Die Trennung erfolgt auf Datenbankebene über Row Level Security (RLS) in Supabase/PostgreSQL. Kein Code im Frontend entscheidet über Datenzugriff.

Row Level Security (RLS): PostgreSQL Policy WHERE user_id = auth.uid() — automatisch für ALLE Queries erzwungen. Tabellen: receipts, profiles, whatsapp_messages, email_inbox, receipt_audit.

🔒 Sicherheit — HMAC Validierung
Jeder Twilio-Webhook wird via HMAC-SHA1 Signatur verifiziert. Der TWILIO_AUTH_TOKEN signiert alle Parameter — gefälschte Requests werden mit HTTP 403 abgelehnt.
👥 Kollegen-Belege
Mitarbeiter die die Sandbox-Nummer gejoined haben, senden Belege an den Inhaber-Account. Unbekannte Nummern bekommen eine Registrierungs-Aufforderung zurück.

06 — DATENSICHERHEIT & DSGVO
🔐

Compliance & Datenschutz

🇪🇺 EU-Hosting & DSGVO
Alle Daten in der EU (Frankfurt, AWS eu-central-1 via Supabase). Kein Drittland-Transfer. AV-Vertrag mit Supabase verfügbar.
🔒 Verschlüsselung
TLS 1.3 für alle API-Verbindungen. Supabase verschlüsselt Daten at-rest (AES-256).
📋 GoBD & §132 BAO
Jede Änderung erzeugt einen Eintrag im receipt_audit-Log. Unveränderlich, nicht löschbar.
🛡 Row Level Security
PostgreSQL RLS erzwingt Mandantentrennung auf Datenbankebene — nicht im Applikationscode.
Wichtiger Hinweis: Klarblick ist ein Vorbereitungs-Tool, kein Ersatz für Steuerberatung. Die finale Prüfung und Einreichung von UVA und Jahresabschluss obliegt dem Steuerberater des Unternehmens.
