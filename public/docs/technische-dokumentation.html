01 — Systemarchitektur
Gesamtüberblick

02 — OCR-Verarbeitung
Wie Belege gescannt werden

Klarblick nutzt Claude Vision (claude-sonnet-4-6) von Anthropic für die optische Zeichenerkennung. Jede Bilddatei oder PDF wird als Base64 direkt an die Claude API übergeben — ohne Drittanbieter-OCR, ohne Umwege.

OCR-VERARBEITUNGS-FLOW — SCHRITT FÜR SCHRITT

Extrahierte Felder

Claude Vision liefert ein strukturiertes JSON mit allen steuerrelevanten Feldern:

// Beispiel-Response von /api/ocr
{
  "vendor":        "Shell Tankstelle Wien",
  "date":           "2026-06-12",
  "gross_amount":   87.40,
  "net_amount":     72.83,
  "vat_amount":     14.57,
  "vat_rate":       20,
  "currency":       "EUR",
  "invoice_type":   "eingang",
  "category":       "Treibstoff/KFZ",
  "receipt_type":   "Tankbeleg",
  "warnings":       []
}
🇦🇹 Österreich-spezifisch
Claude wird explizit mit AT-UStG 1994 instruiert: 20%/13%/10% MwSt-Sätze, Reverse Charge §19, Vorsteuerausschluss §12 (PKW vs. Kastenwagen), ATU-Nummer-Matching.
⚠️ Unsichere Belege
Wenn Claude unsicher ist (unlesbare Schrift, fehlende Felder, ungewöhnliche Beträge), setzt es Einträge im warnings-Array. Der Beleg wird als unsicher markiert.

Eingangs- vs. Ausgangsrechnung — automatische Erkennung

Klarblick erkennt automatisch ob ein Beleg eine Eingangsrechnung (Lieferant → Betrieb) oder eine Ausgangsrechnung (Betrieb → Kunde) ist. Dazu wird die ATU-Nummer des Unternehmens an Claude übergeben:

EINGANGS / AUSGANGS — ERKENNUNGSLOGIK

Confidence Score & Warnings

Status Bedingung Darstellung in App
✓ geprüft Alle Pflichtfelder vorhanden, keine Warnungen Grüner Punkt, sofort verwendbar
⚠ unsicher Claude hat Warnungen gesetzt (unleserlich, fehlende Felder) Gelber Punkt, manuelle Prüfung nötig
✗ fehlgeschlagen Kein JSON extrahierbar (kein Beleg, Foto zu unscharf) Roter Punkt, manuell erfassen
💡 Datenschutz: Beleg-Bilder werden ausschließlich für den OCR-Aufruf an Anthropic übermittelt. Anthropic speichert keine Bilder über den API-Call hinaus (Anthropic API Data Privacy). Die extrahierten Textdaten (Betrag, Lieferant, Datum) liegen ausschließlich in der EU-gehosteten Supabase-Instanz.

03 — Multi-Tenant
Datentrennung & Rollen

Klarblick ist eine Multi-Tenant-Plattform — jeder Betrieb (Tenant) sieht ausschließlich seine eigenen Daten. Die Trennung erfolgt auf Datenbankebene über Row Level Security (RLS) in Supabase/PostgreSQL. Kein Code im Frontend entscheidet über Datenzugriff.

04 — WhatsApp-Integration
Beleg-Upload per Foto

Nutzer senden Belege per Foto an die Klarblick-WhatsApp-Nummer. Twilio empfängt die Nachricht, ruft den Webhook auf, Klarblick liest das Bild per OCR aus und legt automatisch einen Beleg an — ohne dass der Nutzer die App öffnen muss.

WHATSAPP BELEG-FLOW — VOLLSTÄNDIGER ABLAUF
🔒 Sicherheit — HMAC Validierung
Jeder Twilio-Webhook wird via HMAC-SHA1 Signatur verifiziert. Der TWILIO_AUTH_TOKEN signiert alle Parameter — gefälschte Requests werden mit HTTP 403 abgelehnt.
👥 Kollegen-Belege
Mitarbeiter die die Sandbox-Nummer gejoined haben, senden Belege an den Inhaber-Account. Unbekannte Nummern bekommen eine Registrierungs-Aufforderung zurück.

05 — E-Mail-Integration
Rechnungen per E-Mail

Nutzer leiten Rechnungs-E-Mails an ihre persönliche Klarblick-Adresse weiter. Postmark verarbeitet den Eingang, extrahiert Anhänge und sendet alles als JSON an den Webhook. Klarblick führt OCR auf den Anhängen durch und speichert die Ergebnisse.

E-MAIL INBOUND FLOW — POSTMARK INTEGRATION

Webhook-Payload von Postmark

Postmark sendet ein JSON-Objekt mit allen E-Mail-Metadaten und Anhängen (Base64-kodiert):

06 — Datensicherheit & DSGVO
Compliance & Datenschutz
🇪🇺 EU-Hosting & DSGVO
Alle Daten liegen in der EU (Frankfurt, AWS eu-central-1 via Supabase). Kein Drittland-Transfer personenbezogener Daten. AV-Vertrag mit Supabase verfügbar.
🔒 Verschlüsselung
TLS 1.3 für alle API-Verbindungen. Supabase verschlüsselt Daten at-rest (AES-256). Auth-Tokens werden nie im Klartext gespeichert.
📋 GoBD & §132 BAO
Jede Änderung an einem Beleg erzeugt einen Eintrag im receipt_audit-Log (action, field, before_value, after_value, timestamp). Unveränderlich, nicht löschbar.
🛡️ Row Level Security
PostgreSQL RLS-Policies erzwingen Mandantentrennung auf Datenbankebene — nicht im Applikationscode. Selbst ein Bug im Backend-Code kann keine fremden Daten preisgeben.
🔑 API-Schlüssel-Verwaltung
Alle Credentials liegen als Env-Variablen — niemals im Code. SUPABASE_SERVICE_ROLE_KEY und TWILIO_AUTH_TOKEN sind ausschließlich serverseitig verfügbar.
📱 WhatsApp Datenschutz
Twilio Media URLs laufen ab. Bild-Daten werden nur für den OCR-Aufruf verwendet und nicht dauerhaft gespeichert. In whatsapp_messages wird nur die Media-URL, nicht das Bild selbst gespeichert.
